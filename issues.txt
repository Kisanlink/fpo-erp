ISSUE 7: SQLite Migration Compatibility Issues
================================================================================

PROBLEM:
--------
SQLite is failing to create tables with PostgreSQL-specific syntax errors:
- Error: SQL logic error: near "(": syntax error (1)

Affected Models:
- ✗ ProductPrice
- ✗ InventoryTransaction
- ✗ Sale
- ✗ SaleItem
- ✗ DiscountUsage
- ✗ Attachment

Root Causes:
1. timestamptz Type - PostgreSQL-specific (timestamp with timezone)
   SQLite only supports: TEXT, INTEGER, REAL, BLOB, NULL
   Example: `gorm:"type:timestamptz;not null;default:now()"`

2. numeric(12,4) Type - PostgreSQL precision/scale syntax
   SQLite doesn't support precision/scale syntax (12,4)
   The parentheses (12,4) cause the "near '('" syntax error
   Example: `gorm:"type:numeric(12,4);not null"`

3. default:now() Function - PostgreSQL syntax
   SQLite uses: CURRENT_TIMESTAMP or datetime('now')
   Example: `gorm:"default:now()"`

4. default:'INR' with Quotes - Single quotes inside gorm tag
   SQLite may not parse embedded quotes correctly
   Example: `gorm:"default:'INR'"`

5. type:boolean - PostgreSQL boolean type
   SQLite stores booleans as INTEGER (0 or 1)
   Example: `gorm:"type:boolean;not null;default:true"`

Impact:
- Tests fail to create tables for affected models
- Some tests pass because they test error paths (NotFound errors)
- Production is unaffected (uses PostgreSQL which supports all types)

SOLUTION:
---------
RECOMMENDED: Manual SQL Table Creation for SQLite Compatibility

Create a new file: tests/testutils/sqlite_migrations.go

This file will contain SQLite-compatible CREATE TABLE statements for all
affected models, translating PostgreSQL types to SQLite-compatible types.

Type Translations:
- timestamptz → DATETIME
- numeric(12,4) → REAL
- numeric(14,4) → REAL
- numeric(5,2) → REAL
- numeric(10,4) → REAL
- boolean → INTEGER (0 or 1)
- default:now() → CURRENT_TIMESTAMP
- default:'INR' → 'INR' (remove quotes from tag)
- default:true → 1
- default:false → 0

ISSUE 8: GORM Warnings Leaving Blank Spaces in Test Output
================================================================================

PROBLEM:
--------
When running tests with `bash scripts/test-clean.sh go test ./...`, GORM warnings
are being suppressed but blank spaces are left in their place, making test output
hard to read.

Symptoms:
- Test output contains many blank lines where GORM warnings used to be
- Output is cluttered with empty spaces
- Difficult to identify actual test results
- Makes debugging test failures harder

Root Cause:
The GORM logger is using `log.New(io.Discard, "", 0)` which discards the log
content, but GORM may still be formatting log messages (with timestamps, prefixes,
newlines, etc.) even though they're being discarded. The formatting includes
newlines and spaces that are still being printed, leaving blank lines in the output.

```go
        Colorful:                  false,
    },
)
```

Impact:
- Test output is cluttered with blank lines
- Makes it difficult to read test results
- Affects developer experience when running tests

SOLUTION:
---------
RECOMMENDED: Use a Custom Writer That Completely Suppresses Output

Create a custom writer that discards all output including formatting characters,
or use GORM's built-in silent logger more effectively.

ISSUE 11: SQLite Schema Mismatches Causing Test Failures
================================================================================

PROBLEM:
--------
Test failures occur due to SQLite schema mismatches between the model definitions
and the manually created tables in sqlite_migrations.go.

Specific Issues:
1. expiry_date column type mismatch:
   - Model: `ExpiryDate time.Time` with `gorm:"type:date"`
   - SQLite table: `expiry_date TEXT NOT NULL`
   - Should be: `expiry_date DATETIME NOT NULL` for proper date handling

2. Missing columns in inventory_transactions table:
   - Model has: `RelatedEntityID *string` and `PerformedBy *string`
   - SQLite table is missing: `related_entity_id TEXT` and `performed_by TEXT`

Root Causes:
1. Type translation not properly handled - SQLite DATETIME is better than TEXT
   for time.Time fields as GORM can properly serialize/deserialize dates.

2. Manual table creation in sqlite_migrations.go doesn't match the model schema
   exactly, causing INSERT/UPDATE operations to fail.

Impact:
- Tests fail with cryptic errors (made worse by Issue 10)
- Inventory batch creation fails
- Inventory transaction creation fails
- Test debugging is extremely difficult

SOLUTION:
---------
Fix the SQLite migration schema to match the model definitions exactly.

================================================================================

1. Always include actual errors in error messages for better debugging:
   Use: `fmt.Sprintf("Error message: %v", err)` instead of just `"Error message"`

2. SQLite DATETIME type works better with GORM's time.Time serialization than TEXT.
   GORM can properly format dates as ISO8601 strings for DATETIME columns.

3. When manually creating tables for SQLite compatibility, ensure all model fields
   are represented in the table schema, including nullable fields (*string, *int64).

4. Missing columns will cause INSERT/UPDATE operations to fail silently or with
   cryptic errors, making debugging very difficult.
